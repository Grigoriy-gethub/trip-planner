# Описание бизнес-функции: «Интерактивный планировщик маршрута с гибкой оплатой»
## 1. Глоссарий

| Термин | Определение |
| :--- | :--- |
| **Маршрутный конструктор** | Интерфейс, позволяющий пользователю добавлять, удалять и менять порядок элементов путешествия (отели, экскурсии, билеты). |
| **Агент стыковок** | Алгоритм, проверяющий, что время окончания одной активности не перекрывается с началом другой, а расстояние между ними логистически проходимо. |
| **Гибкий платёж (Pay-as-you-go)** | Оплата маршрута не единым платежом, а отдельными частями в разные даты (например, отель сейчас, экскурсии — за 3 дня до старта). |
| **Smart-блокировка** | Временное бронирование мест в отеле или на экскурсию (на 2 часа), чтобы пользователь мог спокойно собирать маршрут. |
| **Логистический модуль** | Сервис, рассчитывающий время и маршрут передвижения между точками маршрута (общественный транспорт, такси, пешком). |

## 2. Бизнес-цели

- **Увеличить среднюю глубину бронирования** (количество услуг на одного клиента) с 1.5 до 3.5.
- **Снизить порог входа** для клиентов, которые не хотят платить полную стоимость отпуска сразу (цель: рост бронирований на 20% за счёт «гибкой оплаты»).
- **Повысить NPS (индекс лояльности)** за счёт прозрачной логистики и отсутствия «сюрпризов» со стыковками.
- **Увеличить частоту визитов в приложение/на сайт**, так как маршрут можно редактировать вплоть до дня поездки.

## 3. Описание бизнес-процесса (BPMN-текст)

**Основной поток (Happy Path):**

1. Пользователь выбирает город и даты поездки.
2. Пользователь переходит в **«Маршрутный конструктор»**:
   - Видит карту города и список доступных «активностей» (отели, музеи, экскурсии, рестораны).
   - Добавляет активности в маршрут простым перетаскиванием (drag-and-drop).
3. **Агент стыковок** автоматически:
   - Проверяет временные окна (музей открыт с 10 до 18, экскурсия начинается в 14:00).
   - Предупреждает, если активности слишком далеко (например, «от Эрмитажа до Исаакиевского собора — 15 минут пешком, вы успеете»).
   - Предлагает оптимальный порядок посещения (по геолокации).
4. Пользователь утверждает маршрут.
5. Система делает **Smart-блокировку** по всем выбранным услугам на 2 часа.
6. На экране оплаты пользователь видит две опции:
   - **«Оплатить всё сейчас»** (единый платёж со скидкой 5%).
   - **«Оплатить по частям»**: сейчас оплачивается только отель и трансфер, а экскурсии и билеты — по отдельному графику (например, за 7 дней до каждой даты).
7. После успешной оплаты (полной или частичной) пользователь получает доступ к маршруту в личном кабинете с push-напоминаниями о предстоящих платежах.

**Альтернативные потоки:**

- **Активность недоступна по времени** → система предлагает альтернативы с похожим рейтингом.
- **Пользователь не успевает по времени** → система автоматически добавляет «буферное время» между активностями (по умолчанию 30 минут) и предупреждает о риске опоздания.
- **Пропуск срока гибкого платежа** → система отправляет 3 напоминания (за 3 дня, за 24 часа, за 2 часа). При неоплате — бронь снимается, а пользователь уведомляется.

## 4. User Story

**Как** путешественник, который любит всё планировать сам, но не хочет промахнуться со временем и логистикой,  
**Я хочу** собрать свой идеальный маршрут из готовых «кирпичиков», где система сама проверит стыковки,  
**Чтобы** получить персонализированный отпуск, в котором не будет сюрпризов, и платить за него удобным для меня способом.

**Критерии приемки (Acceptance Criteria):**

- В конструктор можно добавить не менее 5 типов активностей (отель, музей, экскурсия, ресторан, трансфер).
- Система предупреждает о пересечении времени начала/окончания активностей с точностью до 15 минут.
- Для каждой активности на карте отображается геопозиция.
- При выборе опции «Оплата по частям» система генерирует персонализированный график платежей.

## 5. Ограничения (Constraints)

| Тип | Ограничение |
| :--- | :--- |
| **Технические** | - Карта и геоданные должны загружаться < 2 секунд.
- Время ра


счёта стыковок для маршрута из 7 точек — < 1 секунды.
- Smart-блокировка должна подтверждаться поставщиком за < 3 секунд. |
| **Бизнесовые** | - При гибкой оплате общая стоимость маршрута не может превышать стоимость единой оплаты более чем на 3%.
- Минимальная сумма первого платежа — не менее стоимости отеля + 1 трансфер.
- Гибкая оплата доступна только для маршрутов, стартующих не ранее, чем через 14 дней. |
| **Юридические** | - В договоре оферты должен быть отдельный раздел о графике частичных платежей и последствиях пропуска.
- При частичной оплате права на забронированные экскурсии переходят к клиенту только после полной оплаты конкретной услуги. |
| **Интеграционные** | - API поставщиков должны поддерживать блокировку мест на время (2 часа).
- Требуется интеграция с сервисом геокодинга (2ГИС, Google Maps) для расчёта времени в пути.
- Нужна поддержка подписки на события (webhook) от платёжной системы о статусе частичных платежей. |

## 6. Критерии готовности (DoR — Definition of Ready)

Задача может быть взята в разработку, если:

- [ ] Бизнес-требования и схема «гибкой оплаты» согласованы с финансовым отделом и юристами.
- [ ] Проработаны все варианты стыковок минимум для 10 популярных городов.
- [ ] Утверждён UI/UX для конструктора (особенно для мобильной версии).
- [ ] Определены метрики успешности: конверсия в сборку маршрута > 30%, доля гибкой оплаты > 15%.
- [ ] Доступны тестовые данные поставщиков для всех типов активностей.
- [ ] Команда разработки оценила сложность и подтвердила, что картографический модуль проработан.

## 7. Критерии завершённости (DoD — Definition of Done)

Разработка и релиз функции считаются завершёнными, когда:

- [ ] Написаны unit-тесты для агента стыковок (покрытие > 80%).
- [ ] Проведены интеграционные тесты с каждым типом поставщика (отели, билеты, экскурсии).
- [ ] Выполнено нагрузочное тестирование конструктора: 1000 одновременных пользователей собирают маршрут.
- [ ] Проверена безопасность передачи платёжных данных (PCI DSS).
- [ ] Код прошёл код-ревью и влит в основную ветку.
- [ ] Создана и опубликована документация API для гибких платежей.
- [ ] Проведено A/B-тестирование: группа с конструктором vs группа без него.
- [ ] Настроен мониторинг ошибок в течение 48 часов после релиза.
